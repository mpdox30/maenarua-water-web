# telemetry_gap_fallback_design.md

**สถานะ: เอกสารออกแบบเท่านั้น (2026-07-06) -- ยังไม่ implement เป็นโค้ดจริง ตามที่ผู้ใช้สั่งชัดเจน
("ยังไม่ต้องเขียนโค้ดจริง แค่ร่าง design ไว้ก่อน") รอ sign-off ก่อนเขียนโค้ด**

## ปัญหาที่ต้องแก้

Pipeline รัน scheduled ทุกสัปดาห์ ต้องใช้ water_level รายชั่วโมง (24 ค่า/วัน จาก
telemetry_hourly_aggregate.py) คำนวณ Spill ของแต่ละวัน แต่ telemetry_history_RES002.csv อาจมี
"ช่องว่าง" (is_gap=True บางชั่วโมง) ตรงกับชั่วโมงที่ต้องใช้พอดี -- เกิดได้จากหลายสาเหตุที่พบแล้วจริง
ระหว่างทดสอบ (ดู telemetry_feature.py / telemetry_history_store.py docstring):
  - สถานีหายไปจาก response หลายรอบติดกัน (เช่น RES003 ซ่อมบำรุงตอนนี้ -- RES002 เองก็อาจเกิดแบบ
    เดียวกันในอนาคตได้)
  - non-monotonic response ทำให้ต้อง skip การ append หลายรอบติดกัน
  - เครื่องรัน pipeline offline/พลาดรอบ polling ไปเอง

## แนวทางที่เสนอ: จัดกลุ่มตาม "ขนาด" ของช่องว่าง ไม่ใช่ตัดสินใจแบบเดียวทั้งหมด

หลักการนี้สอดคล้องกับแนวทางที่ใช้มาตลอดในโปรเจกต์นี้ตอนแก้ปัญหา data lineage ของไฟล์ .xlsx
(Group A/B/C: เล็ก/สม่ำเสมอ = ใช้ได้พร้อม flag, ใหญ่/ผิดปกติ = ห้ามเดา ต้อง mark uncertain) -- ใช้
หลักการเดียวกันกับช่องว่างของ telemetry:

### กลุ่ม 1 -- ช่องว่างสั้น (เสนอ: 1-2 ชั่วโมงติดกัน)
**ใช้ water_level ของชั่วโมงใกล้เคียงแทน (interpolate เชิงเส้นระหว่างชั่วโมงก่อน-หลังช่องว่าง)**
เหตุผล: ระดับน้ำอ่างเก็บน้ำเปลี่ยนแปลงช้า (ต่างจาก rainfall ที่เปลี่ยนเร็ว) ช่องว่างสั้นๆ ไม่น่าจะ
พลาดเหตุการณ์สำคัญ (เทียบกับ evap/infiltration bug ที่ขนาดผลกระทบ ~121 m3/day ซึ่งเล็กกว่า noise
ของโมเดลเองมาก -- ระดับน้ำนิ่งในช่วง 1-2 ชม. ก็มีลักษณะคล้ายกัน) **ต้อง flag คอลัมน์ใหม่
`water_level_interpolated=True` ไว้เสมอ เพื่อ audit ย้อนหลังได้ว่าวันไหนใช้ค่า interpolate**
(ตาม pattern เดียวกับ `raw_diverged_from_training_snapshot` ที่ใช้ไปแล้วใน
Training_Values_Nofct_7day_CORRECTED.csv)

**ข้อควรระวังที่ต้องเช็คก่อน interpolate:** ถ้า rainfall_1h ของชั่วโมงก่อน/หลังช่องว่างแสดงว่า
กำลังมีฝนตก (ค่า > 0 อย่างมีนัยสำคัญ) การ interpolate ระดับน้ำเชิงเส้นอาจผิดพลาดมากกว่าปกติ (เพราะ
ระดับน้ำอาจเปลี่ยนเร็วกว่าปกติช่วงฝนตก) -- กรณีนี้ควรลดขนาด "ช่องว่างสั้น" ที่ยอมรับได้ลง (เช่นจาก
2 ชม. เหลือ 1 ชม.) หรือย้ายไปกลุ่ม 2 แทน (ยังไม่ได้กำหนดเกณฑ์ตัวเลขแน่นอน -- ต้องหารือเพิ่ม)

### กลุ่ม 2 -- ช่องว่างยาว (เสนอ: มากกว่า 2 ชั่วโมงติดกัน หรือครอบคลุมทั้งวัน)
**Skip วันนั้นไปเลย พร้อม flag `data_gap_day=True` ในผลลัพธ์ของวันนั้น ไม่เดาค่า**
เหตุผล: สอดคล้องกับหลักการที่ใช้มาตลอดทั้งโปรเจกต์ ("ห้ามคำนวณค่าแทนตรงๆ ให้ mark เป็น
data_lineage_uncertain พร้อมเหตุผล ไม่ตัดสินใจเลือกฝ่ายใดฝ่ายหนึ่งแทนผู้ใช้") -- การ interpolate
ช่องว่างยาวมีความเสี่ยงสูงที่จะได้ค่า Spill ผิดพลาดมาก (พลาดเหตุการณ์ฝนตก/ปล่อยน้ำที่แท้จริงไปทั้งชุด)
แล้วป้อนเข้าโมเดลเป็น "ข้อมูลจริง" โดยไม่รู้ตัว ซึ่งอันตรายกว่าการไม่มีค่าให้เลย (ไม่มีค่า -> เห็นชัดว่า
ต้องระวัง, ค่าเดาผิด -> ดูเหมือนข้อมูลจริงแต่ผิด) ผลลัพธ์: วันนั้นไม่มี Q_in_t ให้ pipeline ใช้
(เหมือนกรณี data_lineage_uncertain ของไฟล์ .xlsx ที่ทำมาก่อนหน้านี้ในโปรเจกต์เดียวกัน)

### เกณฑ์แบ่ง 1-2 ชม. เป็นเพียงข้อเสนอเบื้องต้น ยังไม่ยืนยัน
ค่า threshold ที่แน่นอน (1 ชม.? 2 ชม.? 3 ชม.?) ควรตัดสินใจร่วมกับผู้ใช้ก่อน โดยอาจอ้างอิงจาก:
  - ข้อมูล telemetry_anomalies_RES002.csv ที่กำลังสะสมอยู่ตอนนี้ (scheduled task
    "telemetry-history-poll") -- ดูว่า gap ที่เกิดขึ้นจริงส่วนใหญ่ยาวแค่ไหน สั้น (<1ชม., แค่ non-
    monotonic ธรรมดา) หรือยาว (สถานีหายจริงหลายชั่วโมง)
  - ความสำคัญของ Spill ต่อ Inflow โดยรวม (ถ้า Spill เป็น term เล็กเทียบกับ term อื่นๆ ในสูตร
    water balance ก็อาจยอมรับช่องว่างที่ยาวกว่านี้ได้โดยไม่กระทบมาก -- ยังไม่ได้วัดขนาดเทียบกันจริง)

## สรุปตารางการตัดสินใจที่เสนอ (ยังไม่ implement)

| ขนาดช่องว่าง | การกระทำ | Flag ที่ต้องมี |
|---|---|---|
| 0 ชม. (ไม่มี gap) | ใช้ค่าปกติ | - |
| สั้น (เสนอ 1-2 ชม., ปรับได้) | interpolate เชิงเส้นจากชั่วโมงก่อน-หลัง | water_level_interpolated=True |
| สั้น + มีฝนตกช่วงนั้น | ลดเกณฑ์ลง หรือย้ายไปกลุ่มยาว (เกณฑ์ยังไม่ชัด) | เหมือนกลุ่มที่ย้ายไป |
| ยาว (เกิน threshold) | skip วันนั้น ไม่คำนวณ Spill/Inflow | data_gap_day=True, ไม่มี Q_in_t ให้ใช้ |

## ขั้นตอนถัดไป (รอ sign-off ก่อนเขียนโค้ด)
1. ผู้ใช้ยืนยัน/ปรับ threshold ตัวเลข (1-2 ชม.) และเกณฑ์เรื่องฝนตกช่วง gap
2. เขียน fill_or_flag_gap() ใน telemetry_hourly_aggregate.py (หรือไฟล์แยกใหม่) ตาม
   ตารางการตัดสินใจข้างต้น -- ยังไม่เขียนตอนนี้
3. เชื่อม telemetry_hourly_aggregate.py + fallback logic เข้ากับการคำนวณ Spill จริงใน
   reservoir_reference_data.py (เป็นขั้นตอนแยกต่างหาก หลังข้อ 1-2 เสร็จ)
